# EIMS 追加機能 演習ガイド

## 1. 演習の概要

本演習では、共通機能（社員情報の基本操作）が完成した社員情報管理システム（EIMS）に対し、実務で求められる「セキュリティ」「保守性」「ユーザー利便性」を向上させるための**追加機能**を実装する。

共通機能では詳細な設計書に基づいた開発を経験したが、本演習では提供される「要件」と「画面仕様」から、最適な実装方法を自ら検討し、既存コードへ統合する能力を養う。

### 1.1 開発の目的
- **セキュリティの強化**: 認証機能および権限によるアクセス制御を導入する。
- **データ保全性の向上**: 誤削除防止のための論理削除を導入し、データの復元を可能にする。
- **UI/UX の改善**: 共通レイアウトの導入や視覚情報の追加（顔写真）により、システムの操作性を高める。

### 1.2 実装対象の機能
本演習で追加する機能は以下の通りである。

| 機能 ID | 機能名 | 要件の要点 |
|---|---|---|
| **ADD1** | **ログイン／ログアウト機能** | セッションを用いた利用者認証と、ヘッダーへの状態表示。 |
| **ADD2** | **デザイン改良と共通レイアウト** | 全画面への共通ヘッダー・サイドバーの適用と、検索結果への顔写真表示。 |
| **ADD3** | **退職者管理（論理削除）** | 物理削除の廃止、退職フラグによる管理と、管理者用復元機能。 |
| **ADD4** | **権限による画面操作制御** | 一般ユーザーと管理者の役割に応じた、メニューや操作ボタンの表示制御。 |

### 1.3 前提条件
- **DB スキーマ**: `employee` テーブルには、すでに追加機能用の列（`role`, `delete_flg`）が定義済みである。
- **リソース**: 顔写真データ（`/images/{emp_no}.png`）は所定のディレクトリに配置されているものとする。
- **ベースコード**: 共通機能（検索・登録・変更・削除・詳細）が正常に動作する `EIMS_SpringBoot_Basic` プロジェクトをベースとする。

## 2. 共通仕様・DB構造

追加機能の実装にあたり、以下のデータ定義および共通のルールを遵守すること。

### 2.1 ユーザー権限（ロール）
システム利用者は、以下のいずれかの権限を持つ。

- **一般ユーザー (role = 0)**: 
    - 主に社員情報の参照を行う。
    - 他者の情報を「変更」「削除」することはできない（ボタンを非表示とする）。
- **管理者 (role = 1)**: 
    - システムの全操作が可能。
    - 「全社員一覧」の閲覧や、情報の「変更」「削除」、および「退職者の管理」を行える。

### 2.2 在籍状態（削除フラグ）
社員の在籍状態は、以下のフラグで管理する。

- **在籍 (delete_flg = 0)**: 通常の検索結果に表示されるアクティブな社員。
- **退職 (delete_flg = 1)**: 通常の検索結果からは除外される社員（論理削除状態）。

---

## 3. 機能別要件

### 3.1 ログイン／ログアウト機能 (ADD1)

本機能の目的は、社員番号とパスワードを用いて本人確認を行い、未認証ユーザーによる業務機能へのアクセスを防ぐことである。

#### 3.1.1 認証要件
- **認証キー**: 社員番号（`emp_no`）およびパスワード（`password`）を使用する。
- **ログイン状態の保持**: 認証成功後、ユーザーがブラウザを閉じるか、ログアウトするまで（または一定時間のセッションタイムアウトまで）、ログイン状態を維持すること。
- **アクセス制御**: ログインしていないユーザーが業務メニュー（検索・一覧・登録等）のURLに直接アクセスした場合、ログイン画面へ強制的にリダイレクトさせること。

#### 3.1.2 画面仕様
- **ログイン画面**:
    - 社員番号とパスワードの入力フィールド、およびログインボタンを配置する。
    - 認証に失敗した場合（番号・パスワードの不一致）は、画面上部に「社員番号またはパスワードが正しくありません」という警告を表示すること。
- **ヘッダーの状態表示**:
    - ログイン中：右上に「ようこそ、[姓 名] さん」とログアウトボタンを表示する。
    - 未ログイン：ログイン画面へのリンクを表示する。

---

### 3.2 デザイン改良と共通レイアウト (ADD2)

システムの統一感を高め、情報の視認性と操作性を向上させるための改良を行う。

#### 3.2.1 共通レイアウト（Layout）
- **共通化**: 全てのHTML画面において、ヘッダー、サイドバー、フッターを共通化し、コンテンツ部分のみを差し替える構造を導入すること。
- **サイドバー構成**:
    - トップページ
    - 社員情報検索
    - 社員情報登録
    - （管理者のみ）全社員一覧
- **レスポンシブデザイン**: Bootstrapのグリッドシステムを活用し、PCとタブレット/スマートフォン両方の画面サイズで情報が適切に配置されるようにすること。

#### 3.2.2 検索結果の視覚化
- **顔写真の表示**: 検索結果一覧および詳細画面において、社員番号に対応する画像ファイル（`/images/[emp_no].png`）を表示すること。
    - 画像が存在しない場合の代替えテキスト（Alt）には、社員の氏名を設定すること。
    - 一覧画面での画像サイズは `60px × 70px` 程度に統一し、表のレイアウトが崩れないようにすること。

---

### 3.3 退職者管理・論理削除 (ADD3)

過去の社員データを完全に消去せず、履歴として保持しつつ、通常の業務からは除外する仕組みを導入する。

#### 3.3.1 論理削除の導入
- **削除処理の変更**: 共通機能の「物理削除（DELETE）」を廃止し、「論理削除（UPDATE）」に変更する。
    - 削除実行時は、対象レコードの `delete_flg` を `1` (退職) に更新すること。
- **検索フィルタリング**:
    - 通常の「社員検索」「社員一覧」の検索結果には、`delete_flg = 0` (在籍) の社員のみを表示すること。
    - 退職した社員は、通常の業務画面からは一切見えないように制御する。

#### 3.3.2 退職者の管理（管理者限定）
- **退職者一覧**: 管理者のみがアクセスできる「退職者一覧」画面を新設すること。
- **復元と完全削除**:
    - **復元**: 退職者の `delete_flg` を `0` に戻し、再び通常の検索対象に含める。
    - **完全削除**: 退職者一覧において、本当に不要なデータのみ物理削除（レコードの消去）を可能にする。

---

### 3.4 権限による画面操作制御 (ADD4)

ユーザーの役割（管理者/一般）に応じて、実行できる操作を動的に制限する。

#### 3.4.1 メニュー・導線の制御
- **管理者のみ表示**: 以下の項目は、`role = 1` のユーザーでログインしている場合のみ表示すること。
    - サイドバーの「全社員一覧」リンク
    - サイドバーの「退職者一覧」リンク（ADD3実装時）
- **操作ボタンの制御**: 検索結果一覧における「変更」「削除」ボタンは、管理者のみに表示すること。

#### 3.4.2 詳細画面での制御 (ADD5 統合)
- 社員詳細画面における操作ボタンの表示ルールを以下のように定義する。
    - **編集ボタン**: 「管理者」または「ログインユーザー本人」の場合のみ表示。
    - **削除ボタン**: 「管理者」の場合のみ表示。
- ※一般ユーザーが他人の詳細画面を閲覧した場合は、ボタンが表示されず、情報参照のみが可能となる。

---

## 4. テスト・受け入れ基準

各機能の実装完了後、以下のシナリオが正しく動作することを確認すること。

1. **ADD1**: 未ログインで一覧画面のURLを直接叩くとログイン画面に戻されるか？
2. **ADD2**: PCとスマホで画面を表示した際、サイドバーやテーブルが崩れず表示されるか？
3. **ADD3**: 削除した社員が、通常の検索結果から消え、退職者一覧に現れるか？
4. **ADD4**: 一般ユーザーでログインした際、他人の詳細画面から「変更」「削除」ボタンが消えているか？
5. **ADD5**: ヘッダーの自分の名前をクリックして、自分の詳細画面へ一飛びできるか？
